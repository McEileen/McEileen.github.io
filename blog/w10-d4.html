Tenth Week of RC

Day 5: Testing, Testing

April 25, 2019

Today I submitted a pull request to add the ability to enable, disable, and delete departments from OpenOversight. Admins can do all of the previous behaviors to any department, whereas department area coordinators only have the ability to enable or disable their individual department. Additionally, admins can edit departments, such as by changing the department name or abbreviation. OpenOversight already had some of this existing functionality in place, but it wasn't centralized in a single location. I created a `DepartmentsAPI`, which is displayed in a departments dashboard that only admins can access. Meanwhile, area coordinators can will view a "Department" link within the website's top menu, which takes them to an enable/disable form for their individual department.

I'm proud of myself for writing tests that were thorough and covered a variety of cases. My first job stressed TDD, and I still place a lot of faith in tests. That being said, I want to share some topics that I consider when trying to come up with thorough tests.

  1. Permissions
    • Who is allowed to do what?
    • Are only some users allowed to access routes? Have you confirmed that via tests?
    • Can only some users view certain information? Have you tested that?
    • Is logging in required? Test that.
  2. Data
    • Can you confirm when data has changed?
    • How do data changes affect the application's function?
    • How do data changes alter the application's display?
    • How does the application handle corrupted data?
    • What happens if a user passes malicious data?
    • Have you protected against a possible SQL injection attack?
  3. Existing Tests
    • Do these still pass?
    • Are you sure?
  4. Edge Cases
    • What happens if the number of users spikes?
    • What happens if a "None" or "Null" is entered?
